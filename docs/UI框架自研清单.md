# 游戏内 UI 框架自研清单

## 1. 建设目标

当前项目已有 `project/src/game/ui/theme.rs`、`widgets.rs`、`screens/*` 和 `navigation` 的基础雏形。后续 UI 框架建议定位为一层“游戏 UI 应用框架”，建立在 Bevy UI / ECS 之上，而不是完全重写渲染、布局和输入底层。

核心目标：

- 支持完整的界面生命周期：打开、关闭、隐藏、返回、弹窗叠加、遮罩、界面切换动画。
- 支持鼠标、触控、键盘、手柄的统一输入路由。
- 支持组件化和主题化，减少每个页面重复写 `Node`、`TextFont`、`BackgroundColor`。
- 支持数据驱动布局和样式，让常用页面可以由配置文件声明。
- 支持热加载、调试、性能观测，提升 UI 迭代效率。
- 支持后续联网游戏需要的状态同步、加载中、断线重连、错误提示、权限提示等通用 UI 流程。

非目标：

- 第一阶段不追求实现完整 CSS 引擎。
- 第一阶段不追求可视化编辑器完全替代代码。
- 不把玩法实体、战斗逻辑和 UI 节点强耦合在同一套状态系统里。

### 1.1 模式和界面的关系

当前项目里的 `AppScreen` 名字容易让人误解为“屏幕上唯一存在的全屏 UI”。由于当前项目还没有历史遗留压力，后续 UI 框架设计中建议一步到位改为 App UI Mode。Rust 代码中建议命名为 `AppUiMode`，它表示当前所处的主流程或主模式，用来决定哪些玩法系统、默认 UI 和输入规则启用，而不是表示屏幕上只能存在一个界面。

例如：

- `AppUiMode::Login`：登录流程模式，默认显示登录页。
- `AppUiMode::Lobby`：大厅流程模式，默认显示游戏列表页。
- `AppUiMode::WanfaTouchRipple`：触控水波纹玩法模式，可以没有全屏 UI，只运行触控水波纹场景和必要 HUD。

实际屏幕上同时存在的 UI 由 UI 框架自己的层级和栈管理：

- 页面层：登录页、游戏列表页、玩法 HUD。
- 弹窗层：确认框、设置面板、暂停菜单、网络错误弹窗。
- Toast 层：顶部提示、短消息。
- Loading 层：连接中、匹配中、资源加载中。
- Debug 层：节点树、输入事件流、性能信息。

因此，推荐直接把当前 `AppScreen` 重构为 `AppUiMode`，让它只负责主流程状态；同时新增 `UiPanelId`、`UiPanelKind`、`UiPanelRoot`、`UiPanelCommand`、`UiLayer` 等框架概念来管理共存界面，避免把模式状态和具体 UI 面板混在一起。

## 2. 建议模块结构

后续可以把 `project/src/game/ui/` 扩展为：

```text
project/src/game/ui/
  mod.rs
  framework.rs        # UI 框架插件入口，注册资源、事件、系统
  panel.rs            # Panel Manager：面板注册、面板集合、栈、生命周期和返回
  layer.rs            # UI 层级、弹窗、遮罩、顶层提示
  router.rs           # 主流程 mode 切换、返回、深链参数
  input.rs            # 输入命中测试、焦点、事件路由、输入拦截
  focus.rs            # 键盘/手柄焦点导航
  theme.rs            # 主题 token、颜色、字号、间距、圆角、动效参数
  style.rs            # 样式解析、样式类、状态样式
  layout.rs           # 常用布局构建器和安全区适配
  widgets.rs          # 通用控件
  animation.rs        # UI 动画、过渡、缓动
  assets.rs           # 字体、图片、图集、音效引用管理
  binding.rs          # UI 数据绑定、列表渲染、局部刷新
  i18n.rs             # 国际化文本、参数替换、语言切换
  debug.rs            # 节点树、事件流、焦点、性能调试
  config.rs           # RON/TOML/JSON 配置加载和热刷新
```

`screens/` 继续放具体业务页面，例如登录、房间列表、设置、游戏 HUD。`ui/` 只放框架能力和通用控件。

## 3. 能力清单

### 3.1 界面管理机制

必须有。负责界面层级、显示隐藏、状态切换和返回行为。这里统一使用 Panel 概念：登录页、游戏列表页、玩法 HUD、暂停菜单、设置面板、Loading 遮罩和确认弹窗都属于不同类型的 panel；Toast 保持专用通知系统，不纳入 Panel Manager。

需要支持：

- Panel 注册：用统一标识注册 `LoginPage`、`GameListPage`、`TouchRippleHud`、`TouchRipplePause`、`Settings` 等 panel。
- Panel 生命周期：`on_open`、`on_close`、`on_hide`、`on_show`、`on_pause`、`on_resume`、`on_refresh`。
- Panel 集合和栈：`Page` / `Hud` 常驻，`Floating` 可以多个共存，`Modal` 使用栈，`BlockingOverlay` 通常单例。
- 模态弹窗：确认框、错误框、选择框，支持阻塞下层输入，并参与返回栈。
- Loading 遮罩：作为 `BlockingOverlay` 纳入 Panel Manager，统一处理阻塞输入、关闭和 mode 切换清理。
- 非模态面板：浮动菜单、详情面板、提示条，不阻塞全部输入。
- 顶层系统 UI：Toast、网络状态、调试面板等；Toast 不纳入 Panel Manager，只保留专用通知生命周期。
- 统一清理：mode 退出时自动 despawn 归属该 mode 的 panel 根节点和关联资源。
- 参数传递：打开 panel 时携带参数，例如房间 ID、返回目标、错误码。

建议抽象：

- `UiPanelId`：panel 标识。
- `UiPanelKind`：`Page`、`Hud`、`Floating`、`Modal`、`BlockingOverlay`。
- `UiPanelRoot`：panel 根节点组件，绑定 panel 标识、类型、归属 mode 和层级。
- `UiPanelCommand`：打开、关闭、隐藏、显示、切换、关闭最上层、关闭某个 mode 的全部 panel。
- `UiPanelState`：panel 当前状态。
- `UiLayer`：背景层、页面层、弹窗层、Toast 层、调试层。
- `UiRouteCommand`：打开、关闭、替换、返回。

验收标准：

- 任意 panel 可以通过统一命令打开和关闭。
- 弹窗打开时下层按钮不会误触。
- 返回行为优先处理 `BlockingOverlay`；可取消遮罩关闭，不可取消遮罩消费返回；之后才关闭最上层 `Modal` 或 `Floating` panel，再交给 mode 级返回逻辑。
- mode 切换不会留下孤儿 UI 节点。
- panel 切换可以选择立即切换或播放过渡动画。

### 3.2 UI 组件化

必须有。把按钮、文本、面板、列表等封装成可复用组件，避免页面里重复堆 Bevy UI Bundle。

基础组件：

- 布局：横向布局、纵向布局、自动换行、网格布局、表格布局。
- 文本：标题、副标题、正文、说明、数字、错误文本。
- 按钮：主按钮、次按钮、危险按钮、图标按钮、返回按钮、禁用按钮。
- 面板：普通面板、弹窗面板、列表卡片、HUD 面板。
- 输入：文本输入、数字输入、搜索框、密码框。
- 选择：复选框、单选项、开关、分段控件、下拉菜单。
- 列表：垂直列表、横向列表、滚动列表、虚拟列表。
- 进度：进度条、加载条、倒计时、转圈 Loading。
- 提示：Toast、Tooltip、Badge、红点、状态标签。
- 导航：Tab、面包屑、页面标题栏、底部导航。
- 游戏常用：头像、玩家卡片、房间卡片、道具格、资源条、技能槽。

组件要求：

- 所有组件支持 `normal / hovered / pressed / focused / selected / disabled / loading` 状态。
- 布局组件应优先封装 Bevy UI 的 Flex/Grid 能力，页面代码不应长期直接堆砌重复 `Node` 参数。
- 控件尺寸稳定，状态变化不应造成布局跳动。
- 控件输出统一事件，不直接写业务状态。
- 复杂控件拆成“视图构建”和“行为系统”两部分。

验收标准：

- 新页面 80% 以上 UI 结构可以由通用组件搭出来。
- 按钮视觉、禁用、点击事件不需要每个页面重复实现。
- 列表项可以复用同一个组件渲染不同数据。

### 3.3 主题和样式系统

必须有。当前 `UiTheme` 已经是好起点，后续需要从硬编码主题升级为 token 化和状态样式。

需要支持：

- 颜色 token：背景、面板、边框、文本、主色、危险色、成功色、警告色。
- 字体 token：标题、正文、说明、按钮、数字、等宽字体。
- 间距 token：页面边距、卡片间距、行间距、控件内边距。
- 尺寸 token：按钮高度、输入框高度、图标尺寸、头像尺寸。
- 圆角和边框 token。
- 阴影或层次 token，如后续使用自定义材质或图片背景。
- 状态样式：hovered、pressed、focused、disabled、selected、error。
- 多主题：默认主题、调试主题、高对比主题，后续可扩展活动主题。

建议约定：

- 代码使用语义 token，不直接写具体颜色。
- 页面可以局部覆盖样式，但不能随意绕过主题系统。
- 主题文件可以从 RON/TOML 加载，运行时允许热刷新。

验收标准：

- 修改主题 token 后，大部分页面视觉能统一变化。
- 新增一种按钮样式不需要修改所有页面代码。
- 禁用、选中、聚焦等状态在所有组件上表现一致。

### 3.4 代码与资源分离

很重要。布局、样式、文案和资源引用尽量从业务逻辑中拆出来。

建议分层：

- 代码：处理状态、事件、业务逻辑和复杂交互。
- 配置：描述布局结构、样式类、资源引用、默认文案 key。
- 资源：字体、图片、图标、音效、图集。
- 文案：国际化 key 和多语言文本。

可选配置格式：

- RON：适合 Rust 项目，可读性好，类型表达自然。
- TOML：适合主题 token 和简单配置。
- JSON：适合工具导出或与外部编辑器对接。

建议先支持：

- `assets/ui/themes/default.ron`
- `assets/ui/layouts/*.ron`
- `assets/ui/i18n/zh_cn.ron`
- `assets/ui/i18n/en_us.ron`

验收标准：

- 修改按钮颜色、页面标题、卡片间距不需要改 Rust 代码。
- 配置解析失败时有清晰日志，并能回退到内置默认值。
- 配置文件字段有版本号，后续升级时能兼容或给出错误提示。

### 3.5 输入事件路由

必须有。多界面叠加时，需要明确谁接收点击、谁拦截输入、谁拥有焦点。

需要支持：

- 鼠标点击、移动、滚轮。
- 触控按下、移动、抬起，多指触控预留。
- 键盘确认、取消、方向导航、快捷键。
- 手柄确认、取消、方向导航、肩键切 Tab。
- 命中测试：按层级和 z-order 判断最上层可交互节点。
- 输入吞噬：弹窗、菜单、拖拽控件可以阻止事件继续向下传。
- 焦点系统：记录当前聚焦控件，支持 Tab 或方向键移动焦点。
- 拖拽系统：开始拖拽、拖拽中、结束拖拽、取消拖拽。
- 长按、双击、滑动等组合手势预留。

和当前玩法的关系：

- 当前 `ui_touch` 玩法会捕获屏幕触控并通过 authority 帧同步回放。
- UI 框架需要先判断触控是否命中 UI 控件。
- 未被 UI 消费的输入才进入玩法输入采集。
- 需要记录输入来源，避免 UI 点击和游戏触控同时触发。

验收标准：

- 弹窗按钮点击不会触发底层游戏触控。
- 键盘和手柄可以完整操作登录、列表、设置等页面。
- 当前焦点可视化明确，失焦和禁用状态行为一致。

### 3.6 动画与过渡

应该有。界面切换、按钮反馈、弹窗出现和提示消失都需要统一动效。

需要支持：

- 属性动画：透明度、位置、缩放、尺寸、颜色。
- 生命周期动画：进入、退出、暂停、恢复。
- 页面过渡：淡入淡出、滑入滑出、缩放、遮罩过渡。
- 控件反馈：按钮按下缩放、hover 颜色变化、焦点描边。
- 列表动画：新增、删除、重排时的轻量过渡。
- Toast 和 Loading 的标准动画。
- 缓动函数：linear、ease_in、ease_out、ease_in_out、back、spring 预留。
- 动画完成事件：用于退出后 despawn、切换页面等。

验收标准：

- 页面切换动画不会影响页面生命周期清理。
- 动画中控件是否可点击有明确规则。
- 低性能设备可关闭或降低动效。

### 3.7 数据绑定和局部刷新

必须规划。否则复杂页面会变成手写查询和手动改文本。

需要支持：

- 单向绑定：资源或组件变化后刷新文本、颜色、进度、可见性。
- 列表绑定：根据数据集合生成、复用、删除列表项。
- 条件显示：登录中、已登录、错误、空列表等状态切换。
- 局部刷新：只更新变化节点，避免整页 despawn 重建。
- 空状态、加载状态、错误状态统一表达。

建议抽象：

- `UiBinding<T>`：绑定数据源。
- `UiListView<T>`：列表数据渲染。
- `UiVisibilityRule`：可见性规则。
- `UiTextBinding`：文本绑定，支持 i18n key。

验收标准：

- 房间列表刷新时不会整页闪烁。
- 网络状态变化可以自动更新顶部状态提示。
- 登录错误只更新错误文本和按钮状态。

### 3.8 国际化支持

如果游戏可能面向多语言，应该早期预留。国际化会影响文本长度、布局和资源。

需要支持：

- 文案 key，不在组件中硬编码最终显示文本。
- 参数替换，例如 `{player_name}`、`{count}`。
- 复数和数量格式预留。
- 语言切换后刷新当前页面文本。
- 字体 fallback，不同语言可使用不同字体。
- 文本长度变化时布局不溢出。

验收标准：

- 登录、游戏列表、设置页可以在中文和英文之间切换。
- 缺失文案 key 时有明显日志和兜底文本。
- 长文本不会挤出按钮或覆盖其他 UI。

### 3.9 资源管理和热加载

必须有。UI 迭代需要快速看到效果。

需要支持：

- 字体、图片、图标、图集、音效的统一引用。
- 资源加载状态：未加载、加载中、已加载、失败。
- 占位资源：资源失败时显示默认图或默认样式。
- 主题、布局、文案配置热加载。
- 图片和字体热加载尽量复用 Bevy AssetServer 能力。
- 资源引用检查工具：找出缺失文件和未使用资源。

验收标准：

- 修改主题配置后运行中可刷新。
- 缺图不会导致页面崩溃。
- 启动时能输出 UI 资源加载错误列表。

### 3.10 编辑器工具窗口

如果团队里有设计师或 UI 迭代频繁，这一条很关键。建议分阶段做，不要第一阶段就做完整编辑器。

第一阶段：

- 游戏内调试面板显示 UI 节点树。
- 可以选中节点查看尺寸、位置、样式类、交互状态。
- 可以显示安全区、布局边界、焦点路径。

第二阶段：

- 运行时微调主题 token 和布局数值。
- 支持把调试修改导出为 RON/TOML 片段。
- 支持切换语言、分辨率、输入设备模拟。

第三阶段：

- 独立或游戏内可视化编辑器。
- 拖拽调整布局。
- 组件库面板。
- 与配置文件双向同步。

验收标准：

- 开发者能在运行中定位某个按钮来自哪个配置和哪个实体。
- 修改 UI 间距、颜色后可以快速导出配置。

### 3.11 调试工具

必须有。UI 问题经常来自层级、焦点、输入吞噬和布局尺寸。

需要支持：

- UI 节点树：实体 ID、名称、组件、层级、可见性。
- 布局边界：显示每个节点矩形、padding、border。
- 输入事件流：按下、命中节点、冒泡路径、是否被消费。
- 焦点调试：当前焦点、可导航节点、下一焦点目标。
- Panel Manager 调试：当前常驻 panel、浮动 panel、modal 栈、阻塞 panel、路由历史。
- 样式调试：最终使用的 token、样式类、状态样式。
- i18n 调试：文案 key、实际文本、缺失 key。

验收标准：

- 能快速判断一次点击为什么没触发按钮。
- 能看到当前 panel 栈和阻塞输入的顶层节点。
- 能定位布局溢出的节点。

### 3.12 性能检测机制

必须有。UI 容易因为节点过多、频繁重建、文本过度刷新导致性能问题。

需要观测：

- UI 节点总数和可见节点数。
- 每帧新增、删除、修改节点数量。
- 文本更新次数。
- 图片和字体资源加载数量。
- panel 切换耗时。
- 列表刷新耗时。
- FPS、frame time、系统耗时。
- 过度布局重算和整页重建。

优化方向：

- 列表虚拟化，只生成可见项。
- 对高频变化文本做节流，例如倒计时和网络延迟。
- 避免每帧重建整个 panel。
- 常驻顶层 UI 和业务 panel 分层更新。
- 常用样式缓存，减少重复构造。

验收标准：

- 房间列表 100 条数据滚动稳定。
- panel 切换没有明显卡顿。
- 调试面板能看到 UI 系统耗时和节点数量。

### 3.13 响应式布局和平台适配

必须有。项目有桌面和 Android 打包路径，UI 需要适配不同屏幕和输入方式。

需要支持：

- 分辨率适配：手机竖屏、手机横屏、平板、桌面窗口。
- 安全区：刘海屏、圆角屏、系统导航栏。
- DPI 缩放：字体和控件尺寸按 UI scale 适配。
- 最小点击区域：触控按钮建议不小于 44px 等效尺寸。
- 横竖屏切换。
- 文本过长时换行、省略、缩放或滚动。
- 桌面和移动端可有不同布局配置。

验收标准：

- 登录页、列表页、弹窗在常见手机和桌面尺寸下不溢出。
- Android 触控区域足够大。
- 横竖屏切换后布局能自动恢复。

### 3.14 可访问性和可读性

建议早期考虑基础能力。

需要支持：

- 高对比主题。
- 字号缩放预留。
- 焦点状态清晰可见。
- 重要状态不只依赖颜色区分。
- 错误信息有明确文本。
- 按钮禁用原因可通过 Tooltip 或说明文本表达。

验收标准：

- 仅使用键盘或手柄可以完成核心流程。
- 文本在移动端可读，不依赖过小字号。
- 错误、警告、成功状态有颜色和文字双重表达。

### 3.15 声音和触觉反馈

游戏 UI 建议统一管理反馈，而不是每个按钮自己播放音效。

需要支持：

- 点击、返回、确认、取消、错误、弹窗打开、获得焦点等 UI 音效。
- 触控设备震动或触觉反馈预留。
- UI 音量分类和开关。
- 防重复触发，例如长按或连点不重复播放过多音效。

验收标准：

- 通用按钮点击能自动触发标准音效。
- 禁用、错误、确认等不同操作有可区分反馈。

### 3.16 联网和异步流程 UI

当前项目已有 `authority/` 和 `network/` 模块，UI 框架需要覆盖常见联网状态。

需要支持：

- 全局 Loading：登录中、连接中、匹配中、加载资源中。
- 网络状态条：离线、重连中、已连接、延迟高。
- 错误弹窗：网络错误、登录失败、房间不存在、权限不足。
- 可取消任务：匹配中取消、连接中返回。
- 防重复提交：请求中按钮进入 loading 或 disabled。
- 请求结果统一转 UI 事件。

验收标准：

- 登录和进房流程不会因为连点发送重复请求。
- 网络断开时有明确 UI 状态。
- 错误可以统一弹出并携带可读描述。

### 3.17 测试和示例页面

建议为 UI 框架建立独立样例和回归检查。

需要准备：

- `UiGallery` 页面：展示所有控件和状态。
- `UiStressTest` 页面：大量列表项、频繁刷新、弹窗堆叠。
- `InputTest` 页面：鼠标、触控、键盘、手柄事件流。
- `ThemePreview` 页面：主题 token 预览。
- 基础单元测试：配置解析、路由栈、样式合并、i18n 查找。
- 手动验收脚本：桌面运行、Android 构建、不同分辨率检查。

验收标准：

- 新增控件必须先在 `UiGallery` 可见。
- 修改输入路由后可以在 `InputTest` 验证。
- 修改主题不会破坏基础页面布局。

## 4. 建议开发阶段

### 阶段 0：整理现有 UI 基础

目标：把现有登录、列表和触控页面纳入统一 UI 框架入口。

任务：

- 建立 `UiFrameworkPlugin`，集中注册 theme、widgets、screen、input。
- 明确 `screens/` 和 `ui/` 的职责边界。
- 为所有页面根节点加统一 `UiPanelRoot` 或等价组件。
- 将当前 `AppScreen` 直接重命名并重构为 `AppUiMode`：它只表示主流程，不表示唯一全屏界面。
- 将现有 `Login`、`GameList`、`TouchRipple` 的命名关系拆清楚：`AppUiMode::Login`、`AppUiMode::Lobby`、`AppUiMode::WanfaTouchRipple` 是流程状态；`UiPanelId::LoginPage`、`UiPanelId::GameListPage`、`UiPanelId::TouchRippleHud` 是实际 UI。
- 补充页面层、弹窗层、Loading 层、Toast 层等共存 UI 的层级和 Panel Manager 设计文档。

完成标志：

- 现有页面功能不回退。
- UI 插件入口清晰。
- panel 根节点统一管理和清理。

### 阶段 1：Panel Manager、层级和通用控件

目标：能开发常规游戏菜单和弹窗。

任务：

- 用 `UiPanelId`、`UiPanelKind`、`UiPanelRoot` 替换当前 `UiScreenId`、`UiScreenRoot`。
- 实现 `UiPanelPlugin` 和 `UiPanelCommand`，统一处理 panel 打开、关闭、隐藏、显示、切换、关闭最上层和 mode 切换清理。
- 实现 panel 集合和返回栈：`Page` / `Hud` 常驻，`Floating` 可以多个共存，`Modal` 使用栈，`BlockingOverlay` 通常单例。
- 接入桌面 `Esc` 和 Android Back 到 `UiPanelCommand::CloseTop`，统一返回命令链路。
- 细化 `BlockingOverlay` 的可取消规则：Loading 默认不可取消，显式声明后才能被 `CloseTop` 关闭。
- 在 `UiGallery` 增加 `Floating` 示例 panel，用于验证 `Show Floating`、`CloseTop` 和返回键关闭行为。
- 实现统一按钮、文本、视觉面板、Toast、Loading、确认弹窗。
- 实现最小布局组件：纵向、横向、横向换行、网格。表格可以先作为网格布局的上层约定，后续再补表头、列宽和行数据绑定。
- 实现框架级 ScrollView：滚轮滚动、触控拖动滚动、滚动区域输入拦截。
- Toast 保持专用通知系统，不纳入 Panel Manager。
- Loading 作为 `BlockingOverlay` 纳入 Panel Manager。
- 确认弹窗迁入 Panel Manager，弹窗内容仍可保留独立 `UiModal` 数据结构。
- 实现按钮状态样式和禁用状态。
- 补齐按钮 `focused`、`selected`、`loading` 状态；`loading` 状态按钮默认不触发业务点击，防止重复提交。
- 实现 UI 输入拦截，避免 UI 和玩法触控冲突。

完成标志：

- 登录、游戏列表、玩法 HUD、设置、Loading、确认弹窗可通过统一 panel 命令打开或关闭。
- 弹窗能阻塞下层输入。
- 返回行为优先处理 `BlockingOverlay`，再关闭最上层 `Modal` 或 `Floating` panel。
- `UiGallery` 能直接验证 Floating panel 和 `CloseTop` 行为。
- `UiGallery` 的按钮组和 Overlay 操作区使用框架布局组件，不会因按钮换行导致面板重叠。
- `UiGallery` 正文使用 ScrollView，窗口高度不足时可以滚动查看底部内容。
- 通用按钮能展示 `focused / selected / disabled / loading` 等状态，且禁用和加载中按钮不会触发业务 action。
- 未命中 UI 的触控才进入 `ui_touch` 玩法。

### 阶段 2：主题、配置和热加载

目标：把硬编码视觉参数迁移到主题和配置。

任务：

- 主题 token 从配置文件加载。
- 支持样式类和状态样式。
- 支持文案 key 和基础 i18n。
- 支持主题、文案、布局配置热加载。

完成标志：

- 修改主题配置后可刷新视觉。
- 中英文可以切换。
- 页面代码里的颜色、字号、间距硬编码明显减少。

### 阶段 3：输入、焦点和复杂控件

目标：支持桌面、移动端、手柄完整操作。

任务：

- 实现焦点系统和方向导航。
- 实现文本输入、搜索框、Tab、滚动列表、虚拟列表。
- 实现拖拽、长按、双击等组合输入预留。
- 完成输入事件调试面板。

完成标志：

- 不用鼠标也能操作核心菜单。
- 100 条以上列表滚动稳定。
- 输入事件命中和消费路径可调试。

### 阶段 4：动画、调试和性能工具

目标：提高体验和开发效率。

任务：

- 实现通用 UI 动画系统。
- 实现 panel 过渡和弹窗动效。
- 实现节点树、布局边界、panel 栈、焦点调试。
- 实现 UI 性能统计。

完成标志：

- panel 切换和弹窗出现有统一动效。
- 可以定位布局、输入、焦点问题。
- 能看到节点数量、刷新次数和 UI 系统耗时。

### 阶段 5：编辑器和生产化能力

目标：支持长期内容迭代。

任务：

- 运行时调整主题和布局 token。
- 导出修改后的配置片段。
- 建立 UI Gallery 和压力测试页面。
- 资源引用检查、缺失文案检查、布局溢出检查。
- 视团队需要再做可视化编辑器。

完成标志：

- 设计和开发可以快速试 UI 参数。
- 新增控件有固定展示和验收位置。
- 资源、文案、布局问题可在开发期发现。

## 5. 优先级建议

P0，先做：

- `UiFrameworkPlugin`
- panel 根节点和统一清理
- panel 层级和弹窗层
- 统一按钮、文本、面板、Toast、Loading
- 输入拦截，保护当前 `ui_touch` 玩法
- 主题 token 继续完善

P1，随后做：

- panel 栈和路由参数
- 配置加载和热加载
- i18n 基础能力
- 焦点系统和键盘/手柄导航
- 调试节点树和事件流

P2，成熟后做：

- 复杂控件库
- 虚拟列表
- UI 动画系统
- 性能统计
- 运行时主题调整

P3，按团队规模决定：

- 可视化 UI 编辑器
- 配置导出工具
- 更完整的布局编辑和资源管理工具

## 6. 开发原则

- 先框架骨架，再补控件数量。
- 先覆盖登录、列表、弹窗、设置这些真实页面，再做抽象扩展。
- 控件输出事件，业务系统消费事件，避免控件直接改业务资源。
- 样式用语义 token，不在页面里散落颜色和字号。
- panel 重建和局部刷新要有边界，高频数据不要整页刷新。
- 输入路由必须优先于玩法输入，尤其是当前触控玩法。
- 调试能力要跟框架一起做，不能等 UI 复杂后再补。
- 每个阶段都保留可运行示例页面，避免框架只存在于抽象代码里。

## 7. 最小可用版本定义

UI 框架的最小可用版本建议包含：

- 一个统一 `UiFrameworkPlugin`。
- 一个 Panel Manager 资源。
- 至少三个层级：页面层、弹窗层、Toast 层。
- 通用按钮、文本、面板、Loading、Toast、确认弹窗。
- 统一主题 token。
- UI 输入拦截和焦点预留。
- panel 根节点统一清理。
- 一个 `UiGallery` 示例页面。

达到这个版本后，再继续投入配置化、热加载、复杂控件、编辑器和性能工具，收益会更稳定。
